# Plan: Implementación de integraciones Function Calling vs MCP sobre microservice-kubernetes

## Contexto (hallazgos del análisis del repo)

- Los tres microservicios (`catalog`, `customer`, `order`) exponen controladores Spring MVC que devuelven **HTML** (`ModelAndView`), NO JSON. La API REST real y utilizable por un agente/LLM es la que expone **Spring Data REST** automáticamente sobre los repositorios (`@RepositoryRestResource`), en formato **HAL+JSON**, en las rutas:
  - Catalog: `GET/POST /catalog`, `GET/PUT/DELETE /catalog/{id}`, `GET /catalog/search/findByNameContaining?name=`
  - Customer: `GET/POST /customer`, `GET/PUT/DELETE /customer/{id}`, `GET /customer/search/findByName?name=`
  - Order: `@RepositoryRestResource(path = "order")` declara `GET/POST /order`, `GET/DELETE /order/{id}`, pero **estas rutas están rotas/inalcanzables** (ver hallazgo confirmado abajo).
- Cada microservicio expone su API JSON en su propio puerto 8080 (vía Service/NodePort de K8s), independientemente del Apache reverse-proxy (puerto 8080/80 solo sirve HTML).
- **Hallazgo confirmado (verificado en clúster real, ver `llm-integrations/experiments/baseline_contracts.md`):** en Catalog y Customer, las rutas HTML usan sufijo `.html` (p.ej. `/1.html`), por lo que no colisionan con las rutas JSON de Spring Data REST (`/catalog/{id}`, `/customer/{id}`), y ambas funcionan correctamente. En Order, en cambio, `OrderController` (HTML) mapea rutas **sin sufijo** (`/`, `/{id}`), que colisionan directamente con las rutas que Spring Data REST intentaría exponer en `/order` y `/order/{id}`. Como consecuencia:
  - `GET /order` → HTTP 400 (Spring MVC lo enruta a `OrderController.get(@PathVariable long id)`, intenta parsear el literal `"order"` como `long` y falla).
  - `GET /1` → HTTP 500 (llega a `OrderController.get(1)`, no a Spring Data REST).
  - `GET /profile` → HTTP 400 (el profile de Spring Data REST también queda inalcanzable).
  - **Es decir: hoy no existe ninguna API JSON funcional para Order.** Esto refuerza (y hace obligatoria, no solo recomendable) la tarea `order-json-endpoint`: el nuevo `@RestController` debe vivir en una ruta que no colisione con las rutas existentes de `OrderController` — se usará `/orders` (plural), nunca `/order`.
- El único punto donde `order` valida internamente al `customer` es el flujo HTML (`OrderController` → `OrderService.order()` → `CustomerClient.isValidCustomerId()`), que usa form-encoding, no JSON, y **no valida ítems del catálogo**. Esto es una discrepancia frente al escenario "Alto" descrito en la metodología (que asume que Order valida tanto Catalog como Customer). **Queda pendiente decidir** (ver Puntos de decisión abiertos) si se modifica `OrderService` para añadir validación de ítems, o si la orquestación multi-servicio se resuelve completamente del lado del agente (Function Calling/MCP) con 3 llamadas encadenadas: `GET /catalog/{id}` → `GET /customer/{id}` → `POST /order`.
- Modelos de dominio relevantes para los cambios de esquema (OE1):
  - `Item`: `id`, `name`, `price` (double, flat) — **ambigüedad detectada:** el modelo de dominio de Catalog usa el campo `id`, pero el DTO cliente que usa Order (`com.ewolff.microservice.order.clients.Item`) expone el mismo campo como `itemId`. El cambio de esquema #1 (`itemId` → `id_item`) deberá aclarar explícitamente sobre cuál de las dos representaciones (contrato público JSON de Catalog vs. DTO interno de Order) se aplica el renombre, ya que hoy ya difieren entre sí.
  - `Customer`: `id`, `name`, `firstname`, `email`, `street`, `city`
  - `Order`: `id`, `customerId`, `orderLine[]` (cada línea: `count`, `itemId`) — no tiene `paymentMethod`

## Decisiones ya confirmadas con el usuario

1. **Lenguaje del servidor MCP y del resto del código de integración:** Python (SDK oficial de MCP), reutilizando el mismo lenguaje para Function Calling y el orquestador.
2. **Kubernetes:** el usuario ya tiene minikube/kind/Docker Desktop + kubectl instalados y listos para levantar el clúster; no se incluyen pasos de instalación de estas herramientas, solo verificación y despliegue del proyecto ya existente (`kubernetes-deploy.sh` / `microservices.yaml`).
3. **Acceso a Anthropic:** el usuario aún no tiene API key. El plan incluye una guía breve para obtenerla/configurarla (variable de entorno `ANTHROPIC_API_KEY`), sin asumir que ya existe.
4. **Alcance de esta implementación:** "Implementation-only". Se construyen:
   - Las integraciones Function Calling y MCP (con paridad funcional).
   - La instrumentación de medición (latencia, conteo de llamadas, LOC).
   - Los scripts de escenarios y de cambios de esquema (listos para ejecutar).
   El usuario ejecutará personalmente las 30 corridas cuantitativas y el análisis estadístico final (pandas), por lo que el plan prepara los scripts pero no reporta resultados de ejecución real.
5. **Ubicación del código nuevo:** nuevo directorio de nivel superior en este repo: `llm-integrations/`, con subcarpetas `function_calling/`, `mcp_server/`, `orchestrator/`, `experiments/`.

## Decisión: validación de ítems en creación de pedidos (escenario "Alto")

Se confirmó con el usuario, tras verificar que **ningún endpoint actual encadena una validación de negocio real contra Catalog y Customer a la vez** (solo existe validación de cliente en `OrderService.order()`, y consulta de precio contra Catalog por separado en `Order.totalPrice()`; las llamadas conjuntas en `OrderController`'s `@ModelAttribute` son incidentales, para poblar listas desplegables del formulario HTML, no validación de negocio):

- **Opción elegida: A.** Modificar `OrderService.order()` (Java) para añadir validación de `itemId` contra `CatalogClient` (cambio quirúrgico, junto a la validación de cliente ya existente), de modo que la creación de un pedido dispare un encadenamiento real de llamadas a Catalog y Customer dentro del propio microservicio Order, replicando fielmente el escenario "Alto" descrito en la metodología.
- Esto habilita usar `POST` a un endpoint JSON de creación de pedidos (no el Spring Data REST `POST /order` que no pasa por `OrderService`, sino un endpoint que invoque explícitamente `OrderService.order()`) como la operación "Alto" tanto para Function Calling como para MCP.

## Lista de tareas (todos)

1. **k8s-verify-deploy** — Verificar clúster K8s local activo (`kubectl get nodes`), desplegar los tres microservicios con `kubernetes-deploy.sh`/`microservices.yaml`, y confirmar acceso directo a cada servicio (`kubectl port-forward` a `catalog`, `customer`, `order` puerto 8080) para llamadas JSON.
2. **baseline-endpoints-doc** — Documentar el contrato original (request/response JSON, vía Spring Data REST) de los endpoints seleccionados por escenario: `GET /catalog/{id}` (Bajo), `GET/POST /customer` (Medio). Para Order (Alto), documentar en su lugar el hallazgo de que no existe API JSON funcional hoy (ver arriba) y el contrato objetivo planeado para el nuevo endpoint `/orders`. Guardado en `llm-integrations/experiments/baseline_contracts.md`.
3. **anthropic-access-setup** — Guía y verificación de acceso a la API de Anthropic: crear cuenta/API key, configurar `ANTHROPIC_API_KEY` como variable de entorno, script mínimo de prueba (`ping` a `claude-sonnet-4-6`) en `llm-integrations/orchestrator/verify_api_access.py`.
4. **order-item-validation-fix** — Modificar `OrderService.order()` (Java, microservicio Order) para validar cada `itemId` de las líneas del pedido contra `CatalogClient` (nuevo método, p.ej. `catalogClient.exists(itemId)` o reutilizar `getOne`), junto a la validación de cliente ya existente. Añadir/actualizar tests unitarios existentes de `OrderService`.
5. **order-json-endpoint** — Como las rutas `/order` y `/order/{id}` están inalcanzables (colisión confirmada con `OrderController`, ver hallazgo arriba), crear un endpoint JSON explícito en la ruta **`/orders`** (plural, sin colisión) — `@RestController` con `POST /orders` — que reciba el pedido en JSON, invoque `OrderService.order()` (con la validación de ítems+cliente) y devuelva la respuesta en JSON. Este es el endpoint que usarán las tools de Function Calling y MCP para el escenario "Alto".
6. **function-calling-defs** — Implementar en `llm-integrations/function_calling/` las definiciones de funciones (esquema JSON tipo Anthropic tools) + lógica de invocación HTTP (usando `requests`/`httpx`) para los endpoints seleccionados de Catalog, Customer y Order, con paridad funcional total frente al servidor MCP.
7. **mcp-server-build** — Implementar en `llm-integrations/mcp_server/` un servidor MCP local (Python, SDK oficial), transporte stdio, exponiendo las mismas operaciones como tools con la misma semántica/parámetros que Function Calling, delegando a la misma lógica de invocación REST (reutilizar/compartir un módulo cliente común si es posible sin romper la medición independiente de LOC por arquitectura).
8. **agent-orchestrator** — Construir en `llm-integrations/orchestrator/` el cliente del agente que consume ambas integraciones de forma intercambiable (flag `--mode function_calling|mcp`), manteniendo constantes modelo (`claude-sonnet-4-6`), temperatura y system prompt entre ambas condiciones.
9. **measurement-instrumentation** — Instrumentar logging custom de latencia end-to-end (ms) y conteo de invocaciones a herramientas por tarea, exportando a CSV/JSON, en el propio orquestador (sin alterar la lógica de negocio de los microservicios).
10. **loc-measurement-setup** — Preparar el uso de `cloc` sobre `llm-integrations/function_calling/` vs `llm-integrations/mcp_server/` (+ orquestador compartido) para medir LOC de definición de funciones/tools, lógica de invocación y código de orquestación del agente. Documentar el comando exacto a correr una sola vez al finalizar.
11. **scenario-scripts** — Scripts ejecutables de los 3 escenarios (Bajo/Catalog, Medio/Customer, Alto/Order) parametrizables por arquitectura y número de repeticiones (`n=5`), listos para que el usuario los ejecute (30 corridas totales) y generen los CSV/JSON de latencia y conteo de llamadas.
12. **schema-change-tooling** — Scripts/documentación de Git (branches o parches) para aplicar y revertir, de a uno, los 3 cambios de esquema controlados sobre cada microservicio:
    - Renombrar `itemId` → `id_item` (Catalog)
    - Anidar `price` → `{amount, currency}` (Catalog)
    - Requerir `paymentMethod` nuevo en creación de pedido (Order) — depende de las tareas 4 y 5
    Registrar plantilla para anotar cuántos archivos/líneas se modificaron en cada arquitectura para restaurar funcionalidad (acoplamiento/adaptabilidad, OE1).
13. **integration-parity-check** — Verificación cruzada de paridad funcional: mismas operaciones, semántica y parámetros antes de aplicar cambios de esquema, ejecutando una prueba de humo (smoke test) idéntica contra ambas arquitecturas.

Se detectó y corrigió un bug de bash en la detección del pod de debug recién creado (`grep -vF` con patrón vacío excluía todas las líneas cuando no había pods `node-debugger` previos); reemplazado por una comparación explícita contra un array de nombres de pods previos. Verificado end-to-end contra el clúster real con la variante `payment-method`: `apply.sh payment-method` — reconstrucción/reutilización de imagen, importación a containerd, patch del Deployment y rollout, todo exitoso; `curl POST /orders` sin `paymentMethod` → `400 {"error":"paymentMethod is required"}`, con `paymentMethod` → `201`. `revert.sh payment-method` — deployment vuelve a `microservice-kubernetes-demo-order:local`, `curl POST /orders` sin `paymentMethod` → `201` de nuevo (comportamiento baseline restaurado). El usuario confirmó que esta verificación (1 de 3 variantes) es suficiente; las otras 2 variantes (`itemid-rename`, `price-nested`) comparten exactamente la misma lógica de tooling (ya corregida) y solo difieren en la imagen/parche aplicado.
- ✅ `integration-parity-check` — Completado. Se creó `llm-integrations/scenarios/parity_check.py`, que verifica de forma automatizada (sin depender de comparación manual) la paridad funcional entre Function Calling y MCP con dos chequeos: (1) **paridad de esquema de tools** — compara nombres de tool y nombres de parámetros requeridos entre `function_calling/tools.py` (esquema estilo Anthropic) y el `list_tools()` real del servidor MCP (esquema autogenerado por FastMCP); (2) **paridad de llamadas** — ejecuta el mismo conjunto fijo de casos (get_catalog_item/get_customer válido e inválido, create_order válido e inválido) contra ambas arquitecturas y compara los resultados estructuralmente, ignorando únicamente el campo `id` de `create_order` (no determinista por ser autoincremental en la base de datos — cada llamada crea una fila nueva). Ejecutado por el usuario contra el clúster real (`kubectl port-forward` a los 3 servicios): **5/5 tools con mismo nombre y parámetros, y 6/6 casos de prueba idénticos entre ambas arquitecturas** — confirmando la paridad funcional requerida como línea base antes de aplicar los cambios de esquema (`schema_changes/`). El script queda listo para volver a ejecutarse después de cada `apply.sh <variante>`, donde una divergencia entre arquitecturas es precisamente lo que el experimento OE1 busca medir.

### Adenda — correcciones a `schema-change-tooling` tras iniciar la ejecución de OE1

Durante la ejecución manual del usuario de `apply.sh itemid-rename`, se observó que `parity_check.py` seguía reportando paridad total (sin divergencias) — investigado y explicado: ni `function_calling` ni `mcp_server` parsean campos específicos del JSON (delegan 100% a `common/services.py`, que solo hace *passthrough* del JSON crudo), por lo que un cambio de esquema nunca genera una divergencia *entre* ambas arquitecturas a este nivel — solo podría causar una divergencia si el LLM razona de forma distinta con cada definición de tool (algo que solo se observaría en corridas reales de escenario, no en `parity_check.py`).

Esto llevó a revisar si los **consumidores Java** también debían actualizarse para que el sistema completo (no solo Catalog/Order en aislamiento) siguiera siendo funcional — el usuario confirmó que sí. Se encontraron y corrigieron 2 casos:

- **`itemid-rename`**: Order's `clients/Item.java` (DTO del cliente HTTP hacia Catalog) tenía `@JsonProperty("id")`. Con el renombramiento, Jackson (con `FAIL_ON_UNKNOWN_PROPERTIES=false`) deserializa `itemId` silenciosamente a `0` en vez de lanzar error — no rompía `/orders` (que no lee `getItemId()`), pero sí corrompería la UI HTML de demo (`OrderController.java:62`). Se corrigió a `@JsonProperty("id_item")`, verificado con 15/15 tests y en vivo contra el clúster (orden creada con éxito). Patch: `01b-itemid-rename-order-client.patch`, imagen `microservice-kubernetes-demo-order:schema-itemid-rename`.
- **`price-nested`**: se encontró un problema más grave — Order's `CatalogClient.exists()`/`.price()` deserializan el objeto `Item` completo (incluyendo `price`) vía `RestTemplate`, y solo capturan `HttpClientErrorException` (no errores de deserialización). Verificado en vivo: con `price-nested` aplicado solo a Catalog, `POST /orders` en Order devolvía **HTTP 500** (`MismatchedInputException: Cannot deserialize instance of double out of START_OBJECT token`) — esto violaba el principio guía de que los microservicios deben seguir funcionando. Se corrigió reflejando la misma clase anidada `Price` en Order's `Item.java` (`@JsonIgnore` en los getters/setters flat, `@JsonProperty("price")` en los nuevos `getPriceInfo()`/`setPriceInfo()`). Verificado con 15/15 tests, y en vivo: antes del fix, `POST /orders` → 500; después del fix, → 201. Patch: `02b-price-nested-order-client.patch`, imagen `microservice-kubernetes-demo-order:schema-price-nested`.
- **`payment-method`**: revisado y confirmado que NO sufre este problema — Order es un servicio terminal en el grafo de llamadas (ni Catalog ni Customer llaman a la API de Order, confirmado por `grep`), por lo que no hay un segundo consumidor Java que romper.

`apply.sh`/`revert.sh` se reescribieron para soportar variantes que afectan a más de un microservicio (mapas `SERVICES_FOR`/`IMAGE_FOR`/`PATCH_FOR` ahora indexados por `variante:servicio`), y se verificó end-to-end que tanto `itemid-rename` como `price-nested` aplican/revierten correctamente en ambos servicios (Catalog + Order) sin dejar el sistema en un estado roto. Se creó además `llm-integrations/experiments/oe1_coupling_adaptability.md` como plantilla para que el usuario registre tiempo de adaptación y líneas/archivos modificados (vía `git diff --stat`) por variante y arquitectura, durante la ejecución manual de OE1.
